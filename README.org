* Introduction

nix-hapi ("Nix happy API") is a declarative reconciler for external service
APIs.  It lets you describe the desired state of external services — DNS
records, Git forges, identity providers, LDAP directories, secret managers,
and similar — as pure Nix expressions, then reconciles the live state to
match.

The pipeline is intentionally simple:

#+begin_src
Nix expressions (desired state)
    ↓  nix eval --json
JSON blob
    ↓  nix-hapi apply
Live service calls (per provider)
#+end_src

Nix is the configuration language.  A Rust binary is the reconciler.  There
is no state file: the reconciler always queries live state and diffs against
the expression.

nix-hapi generalises the architecture first validated in
[[https://gitea.proton/logan/ldap-reconciler][ldap-reconciler]] to arbitrary external service providers.

* Installation

** From source

Clone the repository and build with Cargo inside the Nix development shell:

#+begin_src sh :exports code
git clone ssh://git@gitea.proton:2222/logan/nix-hapi.git
cd nix-hapi
nix develop --command cargo build --release
#+end_src

The resulting binary is at ~target/release/nix-hapi~.

** Via Nix flake

#+begin_src sh :exports code
nix run github:LoganBarnett/nix-hapi -- --help
#+end_src

** Including in a consumer flake

Add nix-hapi as a flake input and use its lib, packages, and modules:

#+begin_src nix :exports code
{
  inputs = {
    nixpkgs.url = "github:NixOS/nixpkgs/25.11";
    nix-hapi.url = "github:LoganBarnett/nix-hapi";
  };

  outputs = { self, nixpkgs, nix-hapi, ... }: let
    lib  = nix-hapi.lib;
    ldap = lib.ldap;
  in {
    # Define desired state as a flake attribute.
    nix-hapi-trees = lib.mkFlakeTrees {
      prod-ldap = {
        desiredState = {
          prod-ldap = ldap.mkLdapProvider {
            url          = "ldap://ldap.example.org";
            baseDn       = "dc=example,dc=org";
            bindDn       = "cn=admin,dc=example,dc=org";
            bindPassword = lib.mkManagedFromPath "/run/secrets/ldap-pw";
            users.alice = ldap.mkLdapUser {
              cn           = "Alice Smith";
              sn           = "Smith";
              mail         = "alice@example.org";
              userPassword = lib.mkInitialFromPath "/run/secrets/alice-pw";
            };
          };
        };
        providers.ldap = "${nix-hapi.packages.x86_64-linux.default}/bin/nix-hapi-ldap";
      };
    };

    # NixOS module usage for a host that runs nix-hapi as a service.
    nixosConfigurations.my-host = nixpkgs.lib.nixosSystem {
      modules = [
        nix-hapi.nixosModules.default
        {
          services.nix-hapi = {
            enable = true;
            package = nix-hapi.packages.x86_64-linux.default;
            trees.ldap-service-users = {
              desiredState = { ... };
              providers.ldap = "${nix-hapi.packages.x86_64-linux.default}/bin/nix-hapi-ldap";
              schedule = "daily";
            };
          };
        }
      ];
    };
  };
}
#+end_src

* Usage

** On-demand execution

Evaluate a tree from a consumer flake and pipe it to the reconciler:

#+begin_src sh :exports code
nix eval --json .#nix-hapi-trees.prod-ldap.json \
  | nix-hapi apply --provider ldap=/path/to/nix-hapi-ldap
#+end_src

Or preview changes first with ~plan~:

#+begin_src sh :exports code
nix eval --json .#nix-hapi-trees.prod-ldap.json \
  | nix-hapi plan --provider ldap=/path/to/nix-hapi-ldap
#+end_src

This works from CI, from a developer workstation, or from any host with Nix
installed — no NixOS module required.

** NixOS / nix-darwin service

The ~services.nix-hapi~ module creates a systemd service (or launchd daemon on
macOS) for each declared tree.  This is ideal for host-local reconciliation,
for example maintaining LDAP service users on the same host running the LDAP
server:

#+begin_src nix :exports code
services.nix-hapi = {
  enable = true;
  package = inputs.nix-hapi.packages.${system}.default;
  trees.ldap-service-users = {
    desiredState = {
      local-ldap = ldap.mkLdapProvider {
        url          = "ldap://localhost";
        baseDn       = "dc=example,dc=org";
        bindDn       = "cn=admin,dc=example,dc=org";
        bindPassword = lib.mkManagedFromPath "/run/secrets/ldap-admin-pw";
        users = {
          gitea-service = ldap.mkLdapUser {
            cn           = "Gitea Service Account";
            sn           = "Service";
            mail         = "gitea-svc@example.org";
            userPassword = lib.mkManagedFromPath "/run/secrets/gitea-ldap-pw";
          };
        };
      };
    };
    providers.ldap = "${inputs.nix-hapi.packages.${system}.default}/bin/nix-hapi-ldap";
    # Run daily, or omit for oneshot-after-boot only.
    schedule = "daily";
  };
};
#+end_src

Each tree becomes:
- A JSON file at ~/etc/nix-hapi/<name>.json~.
- A oneshot systemd service ~nix-hapi-<name>.service~ (or launchd daemon).
- An optional systemd timer ~nix-hapi-<name>.timer~ when ~schedule~ is set.

** Plan mode

Show what would change without touching anything:

#+begin_src sh :exports code
nix eval --json .#desiredState | nix-hapi plan
#+end_src

Plan output shows a human-readable diff of additions, modifications, and
deletions that would be applied to each provider.

** Apply

Apply the desired state to the live APIs:

#+begin_src sh :exports code
nix eval --json .#desiredState | nix-hapi apply
#+end_src

** Field management semantics

Fields are managed at the value level.  Every field carries one of three
management markers:

| Marker        | Behaviour                                                     |
|---------------+---------------------------------------------------------------|
| ~mkManaged~   | The reconciler always enforces this value (default).          |
| ~mkInitial~   | Set once when absent; left alone after the first application. |
| ~mkUnmanaged~ | Never touched; documents intentional non-ownership.           |

A plain value is sugar for ~mkManaged~:

#+begin_src nix :exports code
description = "Service account for Gitea";
description = mkManaged "Service account for Gitea";  # equivalent
#+end_src

Runtime-resolved values compose with any marker:

#+begin_src nix :exports code
userPassword = mkManaged (fromPath "/run/credentials/gitea.service/ldap-password");
userPassword = mkInitial (fromPath "/run/credentials/alice.service/ldap-password");
#+end_src

** Ownership and exemptions

nix-hapi assumes it owns everything within the declared scope.  Resources
present in the live API but absent from the Nix expression are candidates for
deletion.  Exemptions are declared via jq filter expressions:

#+begin_src nix :exports code
__nixhapi.ignore = [
  # Each expression receives . = {"key": "<id>", "resource_id": "<id>"}.
  # Truthy output means the resource is exempt from deletion.
  ''.resource_id | test("^uid=placeholder,")''
  ''.key | startswith("_acme")''
];
#+end_src

For file-based expressions, use ~mkJqFile~:

#+begin_src nix :exports code
__nixhapi.ignore = [ (lib.mkJqFile ./ignore-acme.jq) ];
#+end_src

** Cross-provider dependencies and parallel execution

Provider instances that are independent of one another execute concurrently.
Instances that declare a dependency on another instance wait for that instance
to finish before they begin.  The reconciler groups all instances into
sequential /waves/ using a topological sort (Kahn's algorithm), then runs
every instance within a wave in parallel:

#+begin_src
wave 0 │  prod-ldap     prod-dns     ← no dependencies; run in parallel
       │       │              │
wave 1 │  prod-gitea   prod-vault    ← both depend on wave-0 instances
       │        \         /
wave 2 │         prod-app            ← depends on both wave-1 instances
#+end_src

Dependencies are declared with ~__nixhapi.dependsOn~, a list of jq expressions
evaluated against the full top-level JSON blob.  Each expression must resolve
to the scope of the depended-upon instance:

#+begin_src nix :exports code
prod-gitea.__nixhapi.dependsOn = [ ".prod-ldap" ];
prod-app.__nixhapi.dependsOn   = [ ".prod-gitea" ".prod-vault" ];
#+end_src

Because the expressions are jq paths into the Nix-generated JSON, the Nix
layer can compute them from ordinary attribute references — no hard-coded
instance names are required in application code.

** Pre-apply assertions

Assertions are declared in the Nix expression and evaluated against the
computed plan before any API call is made:

#+begin_src nix :exports code
__nixhapi.assertions = [
  { message = "prod records must not be deleted";
    noDestroy = [ "prod\\..*" ];
  }
];
#+end_src

* Gotchas and Tradeoffs

- *No state file*: nix-hapi queries live state on every run.  Reconciliation
  is always a full diff; there is no incremental tracking.

- *Ownership is opt-out*: unknown resources within a managed scope are
  deleted by default.  Use ~__nixhapi.ignore~ patterns to exempt resources
  that are managed outside nix-hapi.

- *~mkInitial~ is stateless*: the reconciler has no memory of whether it was
  the one that set a field.  A field absent from the live API will be reset
  to its initial value on the next run, regardless of why it became absent.

- *Nix is the only input*: the desired state must be fully expressible as a
  Nix value.  Runtime-resolved values (~fromPath~) are resolved by the
  reconciler, not by Nix evaluation.
