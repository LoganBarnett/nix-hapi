* Executor and DAG node-level refactor                            :arch:feat:
  :PROPERTIES:
  :STATUS:   todo
  :END:

  ~derivedFrom~ requires every node in the full JSON tree to be a potential DAG
  vertex, not only top-level provider instances.  This task refactors ~dag.rs~
  and ~executor.rs~ to that model before ~derivedFrom~ is implemented.

** Design

   Current model (provider-scope level):
   - DAG vertices = top-level instance names.
   - Edges come from ~__nixhapi.dependsOn~ jq expressions on each instance.
   - ~execution_waves~ returns ~Vec<Vec<String>>~ of instance names.
   - Executor iterates waves, spawning a thread per instance per wave.

   Target model (arbitrary node level):
   - DAG vertices = absolute jq paths into the full tree (e.g. ~.hr-system~,
     ~.scheduling.memberships.alice-on-call.userId~).
   - Edges come from:
     - ~__nixhapi.dependsOn~ on any ~__nixhapi~ meta block (existing).
     - ~derivedFrom.inputs~ values on any ~FieldValue::DerivedFrom~ node (new).
   - The executor walks the tree to collect all DAG edges, then runs Kahn on
     the full node set.
   - Provider scopes (~.hr-system~, etc.) are still the unit of work: a wave
     entry is a provider scope, not an individual field.  But a ~derivedFrom~
     field in scope A that depends on a field in scope B promotes scope B to
     an earlier wave than scope A would otherwise occupy.
   - Within a wave the executor still spawns one thread per provider instance.

** Acceptance criteria

   - ~execution_waves~ accepts the full JSON value (not ~HashMap<String, Value>~)
     and returns waves of provider-instance names as before, but the wave
     assignment accounts for ~derivedFrom~ inputs at any depth.
   - Adding a ~derivedFrom~ cross-instance input causes the referenced provider
     instance to be placed in an earlier wave, not just the referencing instance
     in a later one.
   - Existing unit tests for ~dag.rs~ all pass.
   - Existing integration tests all pass.

* derivedFrom — runtime-computed field values                          :feat:
  :PROPERTIES:
  :STATUS:   design
  :END:

  Sometimes a field value is not known at Nix evaluation time; it can only be
  computed at reconciliation time from values that earlier parts of the tree
  produced (e.g. a user ID assigned by an HR system that a scheduling system
  needs as a foreign key).

** Agreed design

   New ~FieldValue~ variant:
   #+begin_src rust :exports code
   DerivedFrom {
     inputs: HashMap<String, String>,  // alias → absolute jq path into full tree
     expression: String,               // jq expression; receives $inputs, . is parent scope
   }
   #+end_src

   - ~inputs~ maps a short alias to an absolute jq path into the full top-level
     JSON blob (e.g. ~"user_id": ".hr-system.users.alice.id"~).
   - Each ~inputs~ entry creates an implicit DAG edge: the field that owns this
     ~derivedFrom~ cannot be evaluated until every aliased path resolves.
   - ~expression~ is a jq program where ~$inputs~ is an object of
     ~{ alias: resolved_value, ... }~ and ~.~ is the parent object (the object
     that contains this field).
   - The expression must return a ~FieldValue~-shaped object.  The jq context
     includes injected helpers ~mkManaged~, ~mkInitial~, etc. so the author can
     write ~mkManaged(.user_id)~ instead of raw JSON.
   - ~derivedFrom~ fields always re-evaluate on every reconciliation (semantics
     of ~managed~, not ~initial~).
   - Providers register additional scoped jq helpers (see [[*Provider jq
     functions]]).

   Nix sugar (via ~mkDerivedFrom~):
   #+begin_src nix :exports code
   userId = lib.mkDerivedFrom {
     inputs = { uid = lib.inputPath config.hr-system.users.alice; };
     expression = "mkManaged($inputs.uid)";
   };
   #+end_src

   A good provider ~mk~ function encodes its own ~derivedFrom~ dependencies so
   callers do not need to think about ordering:
   #+begin_src nix :exports code
   # mkScheduleMembership knows that user_id and schedule_id come from
   # their respective providers and declares the inputs itself.
   membership = sched.mkScheduleMembership {
     user   = config.hr-system.users.alice;
     sched  = config.scheduling.schedules.on-call;
   };
   #+end_src

** Prerequisite: node-level DAG refactor

   The current ~dag.rs~ / ~executor.rs~ operate at the provider-instance level
   (top-level JSON keys).  ~derivedFrom~ requires a DAG built from every node
   in the tree, not just provider scopes:

   - DAG nodes are arbitrary jq paths into the full tree.
   - A ~derivedFrom~ field at path ~P~ adds edges ~(inputs[i] → P)~ for each
     alias ~i~.
   - ~dependsOn~ at the provider-instance level becomes a special case of the
     same mechanism (provider scope node depends on another provider scope node).
   - Providers are not special; they are simply at the root by convention.

   This refactor must land before ~derivedFrom~ can be implemented.  See
   [[*Executor and DAG node-level refactor]].

* Shadow values — virtual fields for composition                       :feat:
  :PROPERTIES:
  :STATUS:   design
  :END:

  A shadow (virtual) field is declared in the desired state but the provider
  is instructed to ignore it entirely: it is never read from live state, never
  diffed, and never written.  Its sole purpose is to hold an intermediate value
  that other fields can reference via ~derivedFrom~.

  Example use case: compute a derived DNS record value from the allocated IP
  of a shadow field that holds the cloud provider's output:

  #+begin_src nix :exports code
  allocatedIp = mkShadow (mkDerivedFrom ".cloud.outputs.ip");
  aRecord    = mkManaged "${config.cloud.allocatedIp}";
  #+end_src

  Implementation notes:
  - New ~FieldValue~ variant: ~Shadow { inner: Box<FieldValue> }~.
  - Alternatively, ~Unmanaged~ already means "don't touch", but shadow is
    stronger: the provider should not even read it from live state.
  - Provider ~plan~ implementations must skip shadow fields in both the desired
    scan and the live comparison.
  - The provider trait may need a ~is_shadow(field_name, value)~ helper or a
    pre-processing step that strips shadow fields before the diff.
