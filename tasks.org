* derivedFrom — runtime-computed field values                          :feat:
  :PROPERTIES:
  :STATUS:   design
  :END:

  Sometimes a field value is not known at Nix evaluation time; it can only be
  computed at reconciliation time from values that earlier parts of the tree
  produced (e.g. a user ID assigned by an HR system that a scheduling system
  needs as a foreign key).

** Agreed design

   New ~FieldValue~ variant:
   #+begin_src rust :exports code
   DerivedFrom {
     inputs: HashMap<String, String>,  // alias → absolute jq path into full tree
     expression: String,               // jq expression evaluated with . = resolved inputs
   }
   #+end_src

   - ~inputs~ maps a short alias to an absolute jq path into the full top-level
     JSON blob (e.g. ~"uid": ".[\"hr-system\"].users.alice.id"~).
   - Each ~inputs~ entry creates an implicit DAG edge: the field that owns this
     ~derivedFrom~ cannot be evaluated until every aliased path resolves.
   - ~expression~ is a jq program where ~.~ is the resolved inputs object
     ~{ alias: resolved_value, ... }~.  There is no ~$inputs~ variable — the
     resolved inputs object IS the input, so aliases are accessed as ~.uid~,
     ~.schedule_id~, etc.
   - All input paths must be absolute (from root); no relative addressing.
     The Nix layer generates these paths from attribute references so verbosity
     in the JSON is not a concern for authors.
   - The expression must return a ~FieldValue~-shaped object.  The jq context
     includes injected helpers ~mkManaged~, ~mkInitial~, etc.
   - ~derivedFrom~ fields always re-evaluate on every reconciliation (semantics
     of ~managed~, not ~initial~).
   - Providers register additional scoped jq helpers (see [[*Provider jq
     functions]]).

   Nix sugar (via ~mkDerivedFrom~):
   #+begin_src nix :exports code
   userId = lib.mkDerivedFrom {
     inputs = { uid = lib.inputPath config.hr-system.users.alice; };
     expression = "mkManaged(.uid)";
   };
   #+end_src

   A good provider ~mk~ function encodes its own ~derivedFrom~ dependencies so
   callers do not need to think about ordering:
   #+begin_src nix :exports code
   # mkScheduleMembership knows that user_id and schedule_id come from
   # their respective providers and declares the inputs itself.
   membership = sched.mkScheduleMembership {
     user   = config.hr-system.users.alice;
     sched  = config.scheduling.schedules.on-call;
   };
   #+end_src

** Prerequisite: node-level DAG refactor

   The current ~dag.rs~ / ~executor.rs~ operate at the provider-instance level
   (top-level JSON keys).  ~derivedFrom~ requires a DAG built from every node
   in the tree, not just provider scopes:

   - DAG nodes are arbitrary jq paths into the full tree.
   - A ~derivedFrom~ field at path ~P~ adds edges ~(inputs[i] → P)~ for each
     alias ~i~.
   - ~dependsOn~ at the provider-instance level becomes a special case of the
     same mechanism (provider scope node depends on another provider scope node).
   - Providers are not special; they are simply at the root by convention.

   This refactor must land before ~derivedFrom~ can be implemented.  See
   [[*Executor and DAG node-level refactor]].

* Provider jq functions                                               :feat:
  :PROPERTIES:
  :STATUS:   resolved-by-design
  :END:

  Providers are not aware of each other at runtime — no registration
  mechanism is needed.  Provider authors encode their domain knowledge into
  Nix ~mk*~ helpers (e.g. ~mkScheduleMembership~) that emit the correct
  ~dependsOn~ and ~derivedFrom~ constellations, using ~lib.mkTree~ to
  compute the required jq paths.  Consumers call the helper; the helper
  handles all the wiring.

  The core ~mk*~ functions (~mkManaged~, ~mkInitial~, etc.) are always
  available in every ~derivedFrom~ expression via the ~MK_HELPERS~ preamble
  in ~crates/lib/src/derived.rs~.  No further mechanism is required.

  No implementation work needed.  Close this task.

* Shadow values — virtual fields for composition                       :feat:
  :PROPERTIES:
  :STATUS:   open-question
  :END:

  Open question: do we even need this?

  Since providers are not privileged — any key in the tree outside of a
  provider scope is just data that nobody acts on — a "shadow" value may
  already be achievable by placing a ~derivedFrom~ field somewhere in the
  tree that no provider owns.  The provider only sees its own scope; it
  would never encounter a field declared at an unowned path.

  The original design motivation was intra-provider intermediate values
  (e.g. ~allocatedIp~ hanging off a provider scope, visible to the provider
  but marked "don't diff").  That use case may still require a
  ~FieldValue::Shadow~ variant or a pre-processing strip in the provider
  trait.  Defer until a concrete provider needs it.
